# Product Case Study｜A 股投资辅助决策 Agent

## 1. 项目定位

这是一个面向 A 股短线/波段个人投资者的 AI 决策支持 Agent。它不替用户下单，不承诺收益，也不把 AI 包装成“荐股器”。项目目标是把高噪声、高风险、强时效的交易判断，转化为一套可验证、可降级、可执行、可复盘的决策流程。

核心问题不是“能不能预测涨跌”，而是：

1. 在信息过载时，如何组织证据。
2. 在数据不完整时，如何限制模型输出。
3. 在用户想冲动交易时，如何强制暴露风险。
4. 在判断出错后，如何留下可复盘记录。

## 2. 真实问题与用户场景

目标用户不是缺少行情软件，而是缺少稳定的交易前决策纪律。A 股盘中信息密度高，盘口、题材、公告、板块和持仓盈亏会同时变化，用户很容易从“证据判断”滑向“情绪下注”。

| 维度 | 内容 |
| --- | --- |
| 用户 | A 股短线/波段个人投资者 |
| 高频场景 | 09:28 竞价检查、09:30-09:35 开盘确认、14:30 尾盘计划、单股深研、主题筛选 |
| 输入 | 持仓、观察池、行情、分时、均线、竞价、板块结构、公告新闻、用户手工截图 |
| 决策点 | 买入、持有、加仓、减仓、卖出、观察、取消计划 |
| 风险点 | AI 幻觉、数据滞后、追高、沉没成本、资金代理误读、缺少止损 |
| 成功标准 | 输出可追溯、交易计划完整、缺数据自动降级、复盘样本可沉淀 |

## 3. 为什么适合使用 AI

这个项目不是因为“AI 很热”才使用大模型。原问题中既有结构化规则，也有大量非结构化判断：同一条公告、板块轮动、分时承接、用户持仓心理和交易计划之间存在复杂组合，纯规则难以覆盖。

| 方案 | 优点 | 不足 | 结论 |
| --- | --- | --- | --- |
| 人工看盘和手写计划 | 灵活，贴近经验 | 不稳定，不可审计，容易被盈亏情绪影响 | 不能作为长期稳定流程 |
| 固定规则/Excel | 可控，成本低 | 难处理多因素冲突和自然语言证据 | 适合作为风控底座 |
| 传统搜索/资讯流 | 信息覆盖广 | 只给材料，不会形成结构化计划 | 只能做输入层 |
| 普通大模型问答 | 表达能力强 | 容易编造行情、忽略数据缺口、缺少复盘结构 | 风险过高 |
| Agent + 数据健康门 + 风控规则 | 能组织证据、约束推理、沉淀日志 | 工程和评测成本更高 | 适合本场景的 MVP 方案 |

AI 在这里承担的是“证据组织与推理编排”，不是最终交易决策。规则系统承担数据权限、仓位、止损、输出降级和合规边界；用户保留最终确认权。

## 4. 产品方案

```mermaid
flowchart LR
  A["用户意图 / 持仓 / 观察池"] --> B["数据接入层"]
  B --> C["数据健康门"]
  C --> D["运行包构建"]
  D --> E["AI 推理工作流"]
  E --> F["风控引擎"]
  F --> G["决策计划"]
  G --> H["预测日志 / 结果日志"]
  H --> I["复盘校准"]
```

模块分工：

| 模块 | 作用 | 不交给模型的部分 |
| --- | --- | --- |
| 数据接入层 | 获取行情、分时、均线、市场广度、可选新闻/资金代理数据 | 不允许模型自行补写缺失行情 |
| 数据健康门 | 判断数据等级和输出权限 | 权限规则固定，不由模型临场决定 |
| 运行包构建 | 汇总组合、风险规则、数据缺口和提示词契约 | 上下文范围受控，避免无关信息淹没关键约束 |
| AI 推理工作流 | 按固定顺序组织市场、板块、个股、计划 | 不允许跳过事实/推断/计划分离 |
| 风控引擎 | 由止损距离、账户风险预算、市场状态倒推仓位 | 仓位不是模型信心表达 |
| 复盘校准 | 记录概率、期望 R、实际结果和错误类型 | 不用主观感觉替代样本复盘 |

关键产品取舍见 [Product Decision Record](product_decision_record.md)：Agent 与规则的分工、RAG 使用边界、base rate 来源、风控裁决权和合规边界都在该文档中显式说明。

## 5. 核心工作流

### 09:28 竞价检查

要回答三个问题：

1. 今天市场状态允许新增风险吗。
2. 哪类结构有下注资格。
3. 开盘后哪些信号会确认、取消或降级计划。

关键约束：

1. 缺 A2 竞价层时，不允许说“竞价超预期”，只能给 09:30-09:35 确认条件。
2. 缺 B1 市场层时，市场状态置信度不得高于中。
3. 缺 A0 账户/持仓层时，不能给仓位建议。

### 14:30 尾盘计划

要回答三个问题：

1. 早盘判断哪里对，哪里错。
2. 当前强度是否值得隔夜。
3. 次日 09:28 需要验证什么。

关键约束：

1. 尾盘计划必须扣除隔夜跳空风险。
2. 隔夜计划必须给次日竞价验证条件。
3. 退潮或混沌状态下，默认优先风险管理而不是新增交易。

### 单股深研与主题筛选

要把“概念标签”升级为“证据栈判断”：

1. 单股必须覆盖盘口、K 线/均线、成交、流动性、资金代理、板块角色、催化质量和风险收益。
2. 主题必须区分龙头、中军、补涨、跟风和排除票。
3. 不能因为涨幅大就判定为龙头，必须看启动时间、抗跌性、板块带动和承接。

## 6. 模型不确定性控制

这个项目的重点不是让模型更“自信”，而是让模型在不确定时被系统约束。

| 风险 | 控制机制 |
| --- | --- |
| 幻觉行情、公告或资金流 | 数据层缺失必须标注，缺关键数据自动降级 |
| 把资金流代理当成真实机构意图 | 文档和 Prompt 明确只允许称为 vendor-classified proxy |
| 数据过期仍给盘中动作 | 数据健康门按时间和覆盖率限制输出权限 |
| 用户要求确定性预测 | 输出必须使用概率、条件和失效标准，禁止“必涨/稳赢” |
| 持仓亏损导致加仓冲动 | 贝叶斯更新和沉没成本控制，持仓必须重新评估未来期望 |
| 模型自由发挥 | 固定分析顺序：市场状态、宏观、板块、个股、共振、阶段、操作、风控 |
| 仓位随主观信心放大 | 由 1R、止损距离、账户风险预算和市场状态倒推仓位 |

## 7. 评测与迭代

评测目标不是证明 AI 能预测市场，而是验证它是否遵守数据边界、输出权限和风控纪律。

| 层级 | 指标 |
| --- | --- |
| 模型/输出层 | 数据权限一致性、事实可追溯率、格式稳定性、拒答合理性 |
| 产品层 | 可执行计划率、人工确认率、重试率、计划外交易提示率 |
| 业务层 | 复盘耗时、交易前计划完整率、风险暴露清晰度 |
| 风险层 | 幻觉拦截率、越权输出率、无止损计划率、确定性承诺拦截率 |

完整评测集和失败案例见 [Evaluation Cases And Iteration Notes](evaluation_cases.md)。

公开验证结果见 [Public Validation Report](validation_report.md)。当前仓库提供 30 条离线 guardrail 用例，覆盖数据缺失、RAG 证据边界、风控完整性、用户误用拦截和计划可审计性。验证命令：

```bash
python3 tools/portfolio_validation.py --format markdown
```

该验证不证明投资收益，只证明产品规则和输出边界可以被重复检查。

阈值校准见 [Historical Threshold Calibration](historical_threshold_calibration.md)。过去一个月公开数据覆盖 20 个交易日、100 个 09:28 开盘确认观察和 100 个 14:30 尾盘观察，支持保留 A1 80% 覆盖阈值、B1 市场结构要求和缺 A2 时禁止追强的降级规则。

已识别的关键迭代：

1. 初版只靠 Prompt 约束，模型仍可能直接给交易动作；后续加入数据健康门和输出权限。
2. 初版容易把资金流解释成“主力意图”；后续统一改为资金行为代理，并要求结合价格、成交和板块验证。
3. 初版报告会机械罗列长期不可得的 Tier 3 数据；后续改为只在影响结论时提示。
4. 初版 Demo 偏接口说明；后续改为 Agent 工作台，展示任务、数据健康、输出权限、推理链路和复盘指标。

## 9. 成本、延迟与复杂度取舍

| 取舍 | 当前选择 | 原因 |
| --- | --- | --- |
| 多 Agent 编排 | 暂不引入 | 单次任务链路尚可用固定工作流表达，先保证可审计 |
| 向量库/RAG | 用于消息面证据层，不作为核心行情依赖 | 交易所/公司公告、主流证券报新闻和行情终端数据适合做混合检索与引用；行情、分时和风控仍走结构化规则 |
| 券商交易 API | 明确不接入 | 降低合规风险，保持人在回路 |
| Level-2/筹码等付费数据 | 作为可选层 | 不让不稳定数据成为系统运行前置条件 |
| 前端复杂应用 | 静态 Demo | 作品集展示优先证明机制，不把成本耗在非核心 UI 工程 |
| 实时数据失败 | 降级输出 | 不用模型补数据，避免错误扩大 |

估算口径：

| 项 | 估算方式 |
| --- | --- |
| 人工复盘时间 | 若每天手工整理约 60 分钟，结构化运行包目标是压缩到 20-30 分钟 |
| 计划完整率 | 每个行动项强制包含触发、止损、目标、失效和不交易条件 |
| 幻觉风险 | 以缺数据场景的高置信输出拦截率衡量 |
| 复盘成本 | 预测和结果日志结构化后，可按周聚合错误类型 |

以上为产品测算，不代表投资收益承诺。

## 10. Demo 说明

[互动 Demo](demo/index.html) 不展示代码和命令，而展示 Agent 工作台：

1. 用户选择任务和数据状态。
2. 系统展示数据健康门、数据等级和输出权限。
3. 工作流展示 AI 参与的推理环节，以及规则/风控/人工确认负责的边界。
4. 决策摘要只给“允许输出范围”和“风险约束”，不提供自动交易。
5. 复盘面板展示评测指标、失败类型和校准闭环。

## 11. AI PM 能力映射

| 能力 | 本项目证据 |
| --- | --- |
| 真实问题定义 | 从盘前/盘中交易决策痛点出发，而不是从 AI 功能出发 |
| AI 适用性判断 | 明确 AI 负责证据组织，规则负责权限和风控，用户负责最终确认 |
| RAG/Embedding 理解 | 在消息面证据层设计混合检索、元数据过滤、引用一致性和过期拦截 |
| Agent 工作流设计 | 09:28、14:30、主题筛选、单股深研四类任务 |
| 上下文管理 | 运行包收敛持仓、风控、数据缺口和 Prompt 契约 |
| 模型不确定性控制 | 数据等级、输出权限、缺失降级、拒绝确定性预测 |
| 工程协作意识 | CLI、样例配置、单元测试、GitHub Actions、静态 Demo |
| 评测能力 | 质量指标、风险指标、预测校准、失败样本库 |
| 风险与合规 | 人在回路、无自动下单、无收益承诺、隐私文件不入库 |
| 商业意识 | 复盘时间、计划完整率、风险拦截率和运行成本测算 |

## 12. 下一阶段

1. 接入更稳定的交易日历和实时行情源，降低手工数据依赖。
2. 在已完成过去一个月公开数据阈值校准的基础上，继续补真实 prediction/outcome replay，覆盖概率分桶、Brier score、expected-R 偏差和 50 个消息面查询。
3. 增加用户行为日志：是否按计划执行、是否出现计划外交易。
4. 分离短线、波段、配置型策略，避免不同时间尺度的规则互相污染。
5. 形成一页版和面试讲解版作品集，用同一套证据支撑不同展示深度。
